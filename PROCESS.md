#  JSON 查询 DSL · 解题全过程

## 一、题目初识与学习

一开始拿到这道题目，我并不知道 DSL（领域特定语言）是什么，于是参考了一些资料进行学习：

- [Martin Fowler — Domain Specific Languages](https://martinfowler.com/dsl.html)
- [掘金：《DSL 到底是什么？》](https://juejin.cn/post/7228544909932904504)
- [掘金：《一文搞懂 DSL》](https://juejin.cn/post/7462026326582788132)
- [Elasticsearch Query DSL 官方文档](https://www.elastic.co/docs/explore-analyze/query-filter/languages/querydsl)

**初步认识：这个 DSL 有点像 SQL。** SQL 对关系型数据做 `SELECT ... FROM ... WHERE ...`，而本题要求对 JSON 数据做路径选择（`SELECT`）、数组过滤（`WHERE`）、投影（`SELECT 列`）。本质上就是在写一个 **"JSON 上的 SQL 查询引擎"**。

## 二、题目分析与设计思路

### 2.1 需求拆解

拿到题目后，我将需求分为三个层次：

| 层次 | 功能 | 对应 SQL 概念 |
|------|------|---------------|
| **MVP 基础** | 路径取值 `users[0].name`、数组映射 `users.name` | `SELECT column FROM table` |
| **MVP 核心** | 条件过滤 `users[age > 25]`、操作符 `== != > >= < <=` | `WHERE clause` |
| **加分项** | 管道 `\|`、`contains / startsWith`、安全导航 `?.`、自定义函数 | `ORDER BY`、`LIKE`、`NULL` 安全、`UDF` |

### 2.2 架构思路

类比编译器前端，我设计了一个三阶段管线架构：

```
查询字符串 ──→ [ Lexer ] ──→ Token 流 ──→ [ Parser ] ──→ AST ──→ [ Evaluator ] ──→ 结果
```

这个架构的好处是**关注点分离**：
- **Lexer** 只负责"看字"——把字符流切成有意义的 Token。
- **Parser** 只负责"读句"——把 Token 序列组织成有层次的语法树。
- **Evaluator** 只负责"执行"——拿着 AST 在真实数据上跑。

每个模块可以独立测试、独立改进，不会互相耦合。

### 2.3 关键设计决策

1. **AST 节点设计**：我预先定义了 `Query`、`Path`、`GetField`、`IndexAccess`、`Filter`、`FuncCall`、`BinOp`、`Literal`、`Ident` 共 9 种节点类型。节点种类足够丰富，才能完整表达题目要求的所有查询语义。

2. **递归下降解析**：选择手写递归下降（Recursive Descent）而非第三方库（如 PLY、ANTLR），原因是：
   - 题目语法相对简单，手写更轻量；
   - 面试场景下展现对编译原理的理解更有价值；
   - 无外部依赖，面试官可直接运行。

3. **数组自动映射**：当 `GetField` 遇到 `list` 类型数据时，自动对列表中每个元素提取字段（类似 SQL 的列投影），这是实现 `users.name => ["Alice", "Bob", "Cindy"]` 的关键。

4. **安全导航 `?.`**：在 `GetField` 节点上增加 `safe` 标志位，遇到 `None` 时返回 `None` 而不是抛异常，对标 JavaScript 的可选链（Optional Chaining）。

5. **自定义函数注册机制**：通过 `register_function(name, func)` 实现开放式扩展，用户可以随时注册 `sort`、`take`、`reverse` 等管道函数。

## 三、与 Claude 协作的 Prompt 工程记录

解释说明我与 Claude（Antigravity）协作完成此题的每一步 Prompt，以及各步骤的**目的**。

### Prompt 1：提取题目 + 一次性生成代码与测试

> *"把题目抽取出来放在 bytedance_solution。然后使用 python 实现，并生成10条测试数据跑通，输出测试报告。"*

**目的：** 我的想法是快速建立 MVP。一次性完成 README、核心代码、测试代码的生成，并立即运行验证。先跑通再优化，避免过度设计，而且我在一开始的时候就带着测试，确保后续重构不破坏已有功能。

### Prompt 2：完善 README 的实现目标

> *"你需要在 README 写清楚这个问题的目标，要实现什么方法才能实现这个东西。"*

**目的：** 补充清晰的 README，读的人可以在 30 秒内抓住核心架构：Lexer → Parser → Evaluator。

### Prompt 3：补充词法分析和语法分析的具体实现，加上词法分析和语法分析的结果示例

> *"写清楚你的词法分析、语法分析、求值的具体实现思路和代码，放在 README。要和你的 json_dsl.py 对的上。同时在 README 写一下词法分析、语法分析的结果示例。"*

**目的：** 让README 中的原理说明与实际代码逐行对应，可以交叉验证"文档写的"和"代码实现的"是否一致。对于后续加入这个工程的人来说，可以快速上手，降低阅读门槛。用 Token 列表和 AST 树形图让抽象概念变得直观。

### Prompt 4：按规范仓库结构重组代码

> *"把测试用例单独写成文件，放在 test 目录。把源码放在 src 目录。反正按照规范的代码仓库放你的代码。"*

**目的：** 将代码组织为标准的 Python 项目结构。


### Prompt 5：扩充回归测试用例

> *"多设计一些测试案例，方便你后续改进代码后回归测试。"*

**目的：** 增加测试覆盖率，为后续代码迭代建立安全网。从最初的 11 条扩展到 33 条，覆盖了：边界条件（越界、空数组、负索引）、所有操作符（`!=`, `<`, `<=`, `or`）、错误处理（非法字符、未注册函数）、复合逻辑（括号嵌套 `and/or`）等。一条命令跑完所有用例，确保没有引入新 bug。

### Prompt 6：完善工程化基建（CLI、类型注解、友好错误与 CI）

> *"实现 CLI 命令行入口，支持从文件或 stdin 读取 JSON。给核心代码加上完整的 Type Hints。新增 DSLSyntaxError，用 `^` 精确指出语法错误位置。创建 Makefile 和 GitHub Actions 测试工作流。"*

**目的：** 将算法题级别的玩具代码提升到**工业级可用性**。
- **Type Hints** 降低了团队协作的心智负担，也是 code base 这种基础设施团队的代码底线。
- **友好错误定位**（如编译器般的 `^` 指示符）极大提升了开发者体验（DX），在构建给其他工程师使用的工具时，好用的报错比好用的功能更重要。
- **Makefile & CI** 引入了自动化工程标准，确保每次 PR 都有质量保证。

### Prompt 7：性能认知与基准测试

> *"写一个 benchmark.py，生成 1000 条随机数据，评估各种查询场景的耗时，并测试不同数据规模下的吞吐能力。"*

**目的：** 展现对代码运行代价的把控。作为底层解析引擎，必须清楚 O(n) 遍历在真实规模下的表现。通过 benchmark 验证了 1000 条数据级的复杂过滤查询仅需约 1 毫秒，证明了 AST + 遍历的架构在中小规模场景下是完全可以直接使用的。

### Prompt 8：构建严密的安全网（高级测试策略）

> *"除了回归测试，再实现 4 类测试：Lexer/Parser/Evaluator 的单元测试、CLI 的集成测试、Fuzz Testing（模糊测试）、和 Property-Based Testing（属性测试）。"*

**目的：** 展现对系统稳健性的极致追求，这在维护千万人使用的底层组件时尤为重要。
- **单元测试**让 bug 隔离在组件内部（词法/语法/运行环境）。
- **Fuzz 测试**（模糊测试）通过随机垃圾输入发现了 `str > bool` 比较导致的系统崩溃，促使我修复了该缺陷，证明了“永远不要轻信用户输入”的防范意识。
- **属性测试**（Property Testing）跳出了“输入A得到B”的常规思维，从数学层面验证“数组映射一定不改变长度”、“过滤后的结果集一定是原集合的子集”等恒等性质，这是验证复杂逻辑最有效的高级手段。
